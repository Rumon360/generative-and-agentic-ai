## Positional Encoding in Transformer

Let’s take two sentences — _“Dog ate cat”_ and _“Cat ate dog.”_

If we convert each word into tokens, the vector embeddings generated for both sentences could end up being the same (since they contain the same words).

But in reality, if you understand the meaning, these two sentences are completely different.

This is where **positional encoding** comes in.

So what happens is:

---

### Step 1: Tokenization

Convert words into token IDs.

- _“Dog ate cat”_ → `[56, 74, 89]`

---

### Step 2: Vector Embeddings

Each token ID is converted into a vector.

- `[56, 74, 89]` →

  ```
  Dog → [0.2, 0.8, ...]
  ate → [0.5, 0.1, ...]
  cat → [0.3, 0.7, ...]
  ```

At this stage, both sentences (_“Dog ate cat”_ and _“Cat ate dog”_) will have the **same set of embeddings**, just in different orders.

However, Transformers process tokens in parallel, so they don’t inherently understand the order of words.

---

### Step 3: Positional Encoding

To give the model a sense of order, we add **positional encoding** to each embedding.

This means each position in the sentence gets its own vector, which is added to the word embedding.

- Position 1 → `[p1]`
- Position 2 → `[p2]`
- Position 3 → `[p3]`

Now we combine them:

- Dog (position 1) → `[dog_vector + p1]`
- ate (position 2) → `[ate_vector + p2]`
- cat (position 3) → `[cat_vector + p3]`

For the second sentence:

- Cat (position 1) → `[cat_vector + p1]`
- ate (position 2) → `[ate_vector + p2]`
- Dog (position 3) → `[dog_vector + p3]`

---

### Final Result

Now the vectors are **different**, even though the words are the same, because their positions are different.

- “Dog ate cat” ≠ “Cat ate dog”

The model can now understand:

- who is doing the action
- who is receiving the action

---

### Why Positional Encoding is Important

Without positional encoding:

- The model would treat a sentence like a **bag of words** (no order)

With positional encoding:

- The model understands **sequence and structure**
